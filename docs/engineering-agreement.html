Resumo Executivo

O site é um protótipo visual de altíssima fidelidade com o núcleo transacional de fachada. Este acordo transforma esse núcleo em algo real e honesto: pagamento de verdade (Mercado Pago), RSVP autenticado por link de WhatsApp, regras de segurança que não deixam mais qualquer anônimo forjar uma doação completed, e o fim do eval() de CDN. O trabalho está dividido em 8 SPECs, 2 tracks paralelos e 6 sprints, com verificação em camadas (emulador + sandbox + localhost). Entregável deste ciclo: só o plano; a implementação vem depois.

Decisões Centrais

Q1 · Pagamento

Mercado Pago, por último decidido

Provedor = Mercado Pago (Pix dinâmico + webhooks). Ordem travada pelo noivo: primeiro o catálogo real de presentes (itens com descrição/foto/preço → categorias), depois o pagamento. ADR-0001 (hoje PagSeguro) será atualizada.

Q2 · Guarda de convidado

GuestGate SOFT decidido

O código personaliza e carimba familyId/guestId, mas não bloqueia navegação anônima. Menor risco de travar um convidado real. Endurecimento forte fica para o App Check (depois).

Q3 · Status de pagamento

ENUM de 4 estados decidido

pending · processing · completed · failed. Cliente só cria pending; só o servidor (webhook + Admin SDK) promove. Resolve o "pagamento falho depois de pendente" sem limpeza manual.

Q4 · Autenticação

Anonymous Auth agora decidido

Anonymous Auth dá um uid para carimbar writes e habilitar isSignedIn() nas rules sem fricção. App Check (anti-bot) fica para sprint posterior.

Q5 · Modelo de auth do convidado · o coração da decisão

Capability URL via WhatsApp pessoal decidido

O link entregue no WhatsApp pessoal é a autenticação (magic link). O segredo é um código de alta entropia (8–10 chars, ~40+ bits, inadivinhável) na URL — posse do link = prova de ser convidado. Sem senha, sem login. Recuperação de link perdido = o casal reenvia pelo WhatsApp (consequência aceita do soft gate).

Q6 · Granularidade do código

Um código por família + sub-identificação decidido

Base /rsvp/<código> serve a família inteira. Sem dica → pergunta "quem é você?" (roster real). Link individual opcional /rsvp/<código>?c=<guestId> pré-seleciona a pessoa. Distribuível tanto por família quanto individualmente. A dica é conveniência; a barreira é o código de família.

Q7 · RSVP de família

1 doc por família, qualquer um edita decidido

Um documento em rsvps por familyId. Um membro confirma por todos (roster adults[]); qualquer membro edita depois (updatedBy). Reentrada de quem já confirmou → tela "já confirmado" com resumo (quem confirmou, quando, quem vai) + [Editar resposta] + [Ir para Home/Presentes].

Q8 · Ferramenta de seed

Script admin → códigos + mensagem de WhatsApp decidido

Não há painel. Um script (Firebase Admin SDK, roda local) lê a planilha, gera códigos + guestIds, grava families/guests/codes e imprime a mensagem de WhatsApp pronta por família (link de família + individuais). Disparo automático via WhatsApp Cloud API = SPEC futura opcional.

Dependências & Restrições

  • Endurecer rules antes dos writers — senão derruba RSVP/checkout em produção; mas migrar os writers logo em seguida.
  • Pagamento depende do catálogo (decisão do noivo) e da segurança (só o webhook promove a completed).
  • Gravata depende de pagamento + RSVP-Auth (ranking por família precisa de familyId/guestId).
  • Webhook do MP exige URL pública → túnel (ngrok) ou preview deploy na verificação.
  • domain/types/index.ts é o contrato único de schema — estendido, nunca recriado.
  • Entregável deste ciclo = só o plano (documentação).
